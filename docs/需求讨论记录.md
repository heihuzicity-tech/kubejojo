# K8s 企业级管理系统需求讨论记录

## 1. 当前结论

本项目目标是开发一个基于 React 和 Go 的 K8s 企业级管理系统。

当前已确认的产品范围为：

- 第一阶段定位为 `单集群增强版`
- 交互与页面组织默认参考 `Kubernetes Dashboard`
- 仅在存在明显争议、实现成本过高或与企业需求冲突时，再单独讨论取舍

## 2. 功能清单

### 2.1 第一阶段范围

第一阶段聚焦单集群管理，范围包括以下模块：

1. 工作负载管理
2. Pod 排障与调试
3. 网络资源管理
4. 配置资源管理
5. 存储资源查看
6. 基础监控
7. 权限与审计

### 2.2 详细功能

#### 工作负载

- 使用一个总页面管理工作负载
- 页面内按 `Deployment`、`StatefulSet`、`DaemonSet` 分 `Tab`
- 支持查看、创建、扩缩容、重启、删除
- 支持独立创建页
- 支持独立详情页
- 支持列表直接扩缩容

#### Pod

- Pod 使用独立页面
- 页面定位以排障为主
- 列表直接提供以下快捷入口：
  - 日志
  - 事件
  - `exec terminal`
- 日志支持实时流式刷新
- `exec terminal` 支持选择容器
- Pod 不作为第一阶段的手工创建对象

#### 网络

- 使用一个总页面管理网络资源
- 页面内按 `Service`、`Ingress` 分 `Tab`
- 支持查看、创建、编辑

#### 配置

- 使用一个总页面管理配置资源
- 页面内按 `ConfigMap`、`Secret` 分 `Tab`
- `ConfigMap` 支持查看、创建、编辑
- `Secret` 支持查看、创建、编辑
- 具体交互默认参考 `Kubernetes Dashboard`

#### 存储

- `PVC` 使用独立页面
- 第一阶段以查看为主

#### 监控

- 提供 `概览页`
- 提供 `节点页`
- 支持查看 `Node / Pod` 基础 CPU、内存、状态

#### 权限与审计

- 使用 `Kubernetes Bearer Token` 登录
- 权限模型采用 `RBAC` 透传
- 支持关键操作轻量审计

### 2.3 第一阶段明确不做

- 多集群
- 批量操作
- YAML 在线编辑
- Helm 管理

## 4. 前端技术栈选型

当前前端技术栈结论如下：

- `React`
- `TypeScript`
- `Vite`
- `Ant Design`
- `Ant Design ProTable`
- `Ant Design Form`
- `React Router`
- `TanStack Query`
- `Zustand`
- `Axios`
- `Tailwind CSS`

### 4.1 前端选型原则

- 能使用 `Ant Design` 的地方优先使用 `Ant Design`
- `TanStack Query` 负责服务端状态、缓存、刷新、轮询
- `Zustand` 负责登录态、当前命名空间、筛选条件和 UI 状态
- 界面语言为中文
- 第一阶段不做 `i18n`
- `Tailwind CSS` 作为补充层，不替代 `Ant Design`

## 5. 后端技术栈选型

当前后端技术栈结论如下：

- `Go`
- `Gin`
- 单体服务
- `REST` 为主
- `WebSocket` 用于日志流和 `exec terminal`
- 无数据库
- `client-go + informer/cache`
- `Kubernetes Bearer Token` 认证
- `RBAC` 透传
- 轻量审计日志输出到 `stdout`

### 5.1 后端设计原则

- 后端为单体服务，统一承载认证、K8s 访问、实时代理和审计能力
- 普通资源操作使用 `REST`
- 日志流和终端代理使用 `WebSocket`
- 读操作优先使用 `informer/cache`
- 写操作直接调用 `API Server`
- 第一阶段不引入数据库
- 第一阶段审计采用轻量方案，通过结构化日志输出到 `stdout`

## 6. 整体架构分层

当前确认的系统分层如下：

1. 前端展示层
2. 后端 API 层
3. K8s 访问层
4. 基础能力层

### 6.1 分层职责

#### 前端展示层

- React 页面与路由
- 表格、表单、详情页、终端和日志界面

#### 后端 API 层

- 基于 `Gin` 提供统一接口
- 负责鉴权、参数校验、错误处理、审计

#### K8s 访问层

- 基于 `client-go + informer/cache`
- 读请求走缓存
- 写请求直连 `API Server`

#### 基础能力层

- Token 认证
- RBAC 透传
- 日志流代理
- `exec` 代理
- 审计输出

## 10. 项目目录结构设计

### 10.1 顶层目录

当前已确认使用单仓库管理前后端，顶层结构建议如下：

- `web/`
- `server/`
- `docs/`
- `deploy/`
- `scripts/`

### 10.2 顶层目录职责

- `web/`：React 前端项目
- `server/`：Go 后端项目
- `docs/`：需求、设计、接口等文档
- `deploy/`：部署清单与 Kubernetes 相关配置
- `scripts/`：本地开发、构建、检查等脚本

### 10.3 前端目录原则

- 前端业务代码按 `src/modules` 分模块组织

### 10.4 后端目录原则

- 后端采用 Go 官方常见目录组织方式：
  - `cmd/`
  - `internal/`
  - `pkg/`
- 程序主入口按 `cmd/<app>/main.go` 组织

### 10.5 默认决策原则

- 后续未明确的小问题，优先采用官方最佳实践
- 在不影响当前产品目标的前提下，尽量减少重复确认

### 10.6 前端 `web/src` 目录建议

前端建议采用按职责分层、按业务模块归档的目录结构：

- `src/app/`
- `src/router/`
- `src/layouts/`
- `src/pages/`
- `src/modules/`
- `src/components/`
- `src/services/`
- `src/stores/`
- `src/hooks/`
- `src/utils/`
- `src/types/`
- `src/assets/`
- `src/styles/`

各目录职责建议如下：

- `src/app/`：应用启动、全局 Provider、QueryClient、全局配置
- `src/router/`：路由定义与路由守卫
- `src/layouts/`：登录后整体布局、顶部状态区、侧边导航
- `src/pages/`：页面级入口，负责组装模块页面
- `src/modules/`：按资源域组织的业务模块，是前端核心目录
- `src/components/`：跨模块复用组件
- `src/services/`：基于 `Axios` 的接口封装
- `src/stores/`：基于 `Zustand` 的全局状态
- `src/hooks/`：通用 Hooks
- `src/utils/`：工具函数
- `src/types/`：共享类型定义
- `src/assets/`：静态资源
- `src/styles/`：全局样式、主题补充、Tailwind 入口

#### `src/modules/` 建议模块

- `modules/auth/`
- `modules/overview/`
- `modules/nodes/`
- `modules/workloads/`
- `modules/pods/`
- `modules/network/`
- `modules/config/`
- `modules/storage/`
- `modules/audit/`

#### 前端目录补充原则

- 页面入口放在 `pages/`，具体业务实现放在 `modules/`
- 公共能力优先沉淀到 `components/`、`hooks/`、`services/`
- 尽量避免把复杂业务逻辑堆积到页面文件中

### 10.7 后端 `server/internal` 目录建议

后端建议采用按领域和基础能力拆分的结构：

- `internal/app/`
- `internal/config/`
- `internal/server/`
- `internal/middleware/`
- `internal/handler/`
- `internal/service/`
- `internal/kube/`
- `internal/ws/`
- `internal/audit/`
- `internal/model/`
- `internal/response/`

各目录职责建议如下：

- `internal/app/`：应用装配与启动流程
- `internal/config/`：配置加载与配置结构
- `internal/server/`：Gin 路由注册、HTTP 服务启动
- `internal/middleware/`：认证、审计、错误处理、中间件
- `internal/handler/`：HTTP / WebSocket 接口处理层
- `internal/service/`：业务编排层
- `internal/kube/`：`client-go`、`informer/cache`、资源访问封装
- `internal/ws/`：日志流和 `exec terminal` 的 WebSocket 处理
- `internal/audit/`：审计事件定义与结构化输出
- `internal/model/`：内部领域模型与 DTO
- `internal/response/`：统一响应格式与错误码

#### `internal/service/` 建议领域划分

- `service/auth`
- `service/overview`
- `service/nodes`
- `service/workloads`
- `service/pods`
- `service/network`
- `service/config`
- `service/storage`
- `service/audit`

#### 后端目录补充原则

- `handler` 只处理协议层，不堆积复杂业务逻辑
- `service` 负责业务编排
- `kube` 统一封装与 Kubernetes 的交互
- 读操作优先走缓存，写操作直接调用 `API Server`

## 11. 前后端接口清单设计

### 11.1 设计原则

- 第一阶段接口以 `REST` 为主
- 实时能力通过 `WebSocket` 提供
- 接口按资源域组织
- 单集群前提下，接口路径仍显式传递 `namespace`
- 写操作由后端统一执行鉴权校验、K8s 调用和审计记录

### 11.2 统一约定

#### 接口前缀

- HTTP 接口统一使用 `/api/v1`
- WebSocket 接口统一使用 `/ws/v1`

#### 通用查询参数

列表类接口默认支持以下常见参数：

- `namespace`
- `page`
- `pageSize`
- `search`
- `labelSelector`

说明：

- 第一阶段以前端统一切换 `namespace` 为主
- 后端接口仍显式接收 `namespace`
- 不同资源可按实际情况增减筛选参数

### 11.3 认证与基础信息

#### 认证

- `POST /api/v1/auth/login`
  - 用途：提交 `Bearer Token` 并完成可用性校验
- `GET /api/v1/auth/me`
  - 用途：获取当前登录身份信息
- `POST /api/v1/auth/logout`
  - 用途：前端清理会话态

#### 基础上下文

- `GET /api/v1/namespaces`
  - 用途：获取当前用户可见的命名空间列表
- `GET /api/v1/overview`
  - 用途：获取概览页聚合数据

### 11.4 节点与概览

#### 节点

- `GET /api/v1/nodes`
  - 用途：获取节点列表
- `GET /api/v1/nodes/{name}`
  - 用途：获取节点详情
- `GET /api/v1/nodes/metrics`
  - 用途：获取节点基础指标

#### 概览

- `GET /api/v1/overview/summary`
  - 用途：获取概览页摘要卡片数据
- `GET /api/v1/overview/pods/abnormal`
  - 用途：获取异常 Pod 摘要
- `GET /api/v1/overview/metrics`
  - 用途：获取资源使用率数据

### 12.5 页面职责建议

#### 登录页

- 输入 `Bearer Token`
- 完成登录校验
- 登录成功后进入概览页

#### 概览页

- 展示集群状态
- 展示节点状态
- 展示资源使用率
- 展示命名空间资源统计
- 展示异常 Pod 摘要

#### 节点页

- 展示节点列表
- 支持查看节点详情
- 展示节点基础状态与资源使用情况

#### 工作负载页

- 单页面承载工作负载管理
- 通过 `Tab` 区分 `Deployment / StatefulSet / DaemonSet`
- 提供搜索、筛选、创建入口、扩缩容、重启、删除

#### Pod 页

- 面向排障与调试
- 展示 Pod 状态、容器信息、所属工作负载、所在节点
- 提供日志、事件、终端入口

#### 网络页

- 通过 `Tab` 区分 `Service / Ingress`
- 支持查看、新建、编辑

#### 配置页

- 通过 `Tab` 区分 `ConfigMap / Secret`
- 支持查看、新建、编辑

#### PVC 页

- 展示 PVC 列表与详情
- 第一阶段以查看为主

#### 审计日志页

- 展示关键操作审计记录
- 支持按时间、资源类型、操作类型筛选

### 12.6 页面交互建议

#### 列表页

- 使用 `ProTable`
- 默认展示状态、名称、命名空间、创建时间、关键属性
- 常用操作直接放到操作列

#### 创建页

- 独立页面承载创建流程
- 优先使用 `Ant Design Form`
- 提交后跳转到对应详情页或列表页

#### 详情页

- 展示基础信息、状态信息、关联资源和关键操作
- 结构与 `Kubernetes Dashboard` 保持一致性优先

### 12.7 路由守卫与上下文建议

- 未登录用户统一跳转 `/login`
- 登录成功后默认跳转 `/overview`
- 路由切换过程中保留当前 `namespace`
- 页面刷新后应尽量恢复当前登录态与 `namespace`

## 13. MVP 开发里程碑拆分

### 13.1 拆分原则

- 先完成项目骨架与基础链路
- 再完成核心资源管理能力
- 再补齐排障、监控和审计
- 优先交付“能登录、能看、能管、能排障”的闭环

### 13.2 里程碑规划

#### M0：项目初始化

目标：

- 完成前后端项目初始化
- 建立基础目录结构
- 建立本地开发和联调方式

交付内容：

- `web/` 初始化完成
- `server/` 初始化完成
- 基础目录结构创建完成
- 基础文档建立完成
- 前后端开发环境可启动

#### M1：基础骨架打通

目标：

- 打通登录、路由、布局、基础 API 链路

交付内容：

- `Bearer Token` 登录页
- 前端路由与全局布局
- 顶部 `namespace` 与用户信息区域
- 后端 `Gin` 服务启动
- 基础认证中间件
- `/auth/login`、`/auth/me`、`/namespaces`、`/overview` 等基础接口

完成标志：

- 用户可登录系统
- 可获取当前身份信息
- 可切换命名空间
- 可进入概览页

#### M2：工作负载管理

目标：

- 完成交付价值最高的工作负载管理模块

交付内容：

- 工作负载总页面
- `Deployment / StatefulSet / DaemonSet` 三个 `Tab`
- 列表查询
- 创建页
- 详情页
- 扩缩容
- 重启
- 删除

完成标志：

- 用户可完成主要工作负载的查看与基础运维操作

#### M3：Pod 排障能力

目标：

- 建立排障闭环

交付内容：

- Pod 列表页
- Pod 详情页
- 事件查看
- 日志实时流
- `exec terminal`
- 容器选择能力

完成标志：

- 用户可从 Pod 列表完成常见排障操作

#### M4：网络、配置、存储

目标：

- 补齐常用资源管理能力

交付内容：

- 网络页
- `Service / Ingress` 查看、创建、编辑
- 配置页
- `ConfigMap / Secret` 查看、创建、编辑
- PVC 列表与详情页

完成标志：

- 用户可完成常见网络与配置资源维护

#### M5：监控与审计

目标：

- 补齐基础可观测性与操作追踪能力

交付内容：

- 概览页摘要卡片
- 节点页
- Node / Pod 基础 CPU、内存、状态
- 审计日志输出
- 审计日志页

完成标志：

- 用户可查看基础运行状态
- 关键操作可追踪

#### M6：收尾与可交付

目标：

- 统一补齐工程化与部署交付能力

交付内容：

- 错误处理统一化
- 空态、异常态、加载态补齐
- 接口与权限边界检查
- 部署配置整理
- 使用说明文档

完成标志：

- 系统具备内测交付条件

### 13.3 推荐实施顺序

建议严格按以下顺序推进：

1. M0 项目初始化
2. M1 基础骨架打通
3. M2 工作负载管理
4. M3 Pod 排障能力
5. M4 网络、配置、存储
6. M5 监控与审计
7. M6 收尾与可交付

### 13.4 当前最小闭环定义

在当前产品定位下，最小可用闭环定义为：

1. 用户可通过 `Bearer Token` 登录
2. 用户可切换命名空间
3. 用户可查看工作负载
4. 用户可创建和管理主要资源
5. 用户可查看 Pod 日志并进入终端
6. 用户可查看基础运行状态
7. 用户关键操作可被审计
