# K8s Admin 产品方案与需求基线

## 1. 文档说明

- 本文档用于沉淀当前已经确认的产品范围、技术路线、页面信息架构和开发边界。
- 文档会明确区分“已确认规划”和“当前已落地状态”，避免把未来设想与当前代码现状混在一起。
- 未单独讨论的小问题，默认参考 `Kubernetes Dashboard` 与 `Headlamp` 的官方最佳实践：
  - 功能分组与资源域导航优先参考 `Headlamp`
  - 总览页、运维导向和信息密度优先参考 `Kubernetes Dashboard`
- 视觉层不直接照抄官方产品，但交互优先服从稳定性、清晰层级、信息密度和运维效率。

## 2. 产品定位

- 项目名称：`K8s Admin`
- 产品定位：面向企业内部运维场景的 `Kubernetes` 单集群管理系统
- 一期定位：`单集群增强版`
- 当前运行方式：前后端运行在本机开发机上，直接连接真实实验集群
- 默认语言：中文
- 第一阶段不做 `i18n`

## 3. 已确认的核心决策

- 认证方式：`Kubernetes ServiceAccount Bearer Token`
- 权限模型：直接复用 `Kubernetes RBAC`
- 前端：`React + TypeScript + Vite + Ant Design`
- 状态管理：`TanStack Query + Zustand`
- 列表组件：优先使用 `Ant Design ProTable`
- 请求库：`Axios`
- 样式补充：`Tailwind CSS`
- 后端：`Go` 单体服务
- Web 框架：`Gin`
- 接口风格：以 `REST` 为主
- 实时能力：`WebSocket` 用于日志流和 `exec terminal`
- 数据存储：第一阶段不落库
- 审计：保留轻量审计，先输出到 `stdout`
- 设计取向：保留 `Ant Design` 组件能力，但 `Layout / Navigation / Overview` 采用自定义骨架，不使用默认后台模板风格

## 4. 一期范围

### 4.1 功能域

- 集群管理
- 资源全景图
- 工作负载
- 网络管理
- 存储管理
- 安全管理
- 配置管理
- 资源管理

### 4.2 各模块目标能力

#### 集群管理

- `Overview`：总览页，提供集群健康、资源使用、告警事件、节点状态、命名空间资源分布
- `Namespaces`：命名空间列表、状态、标签与详情入口
- `Nodes`：节点列表、健康状态、版本、资源与详情入口

#### 资源全景图

- `Topology`：以关系画布为中心展示资源依赖与访问链路
- 初版以真实集群资源关系展示为主，不做纯展示型假数据拓扑

#### 工作负载

- `Pods`
- `Deployments`
- `StatefulSets`
- `DaemonSets`
- `ReplicaSets`
- `Jobs`
- `CronJobs`
- 目标能力：查看、详情、创建、编辑、删除
- `Pods` 额外保留排障能力：日志、事件、`exec terminal`

#### 网络管理

- `Services`
- `Endpoints`
- `Ingresses`
- `IngressClasses`
- `NetworkPolicies`
- `Gateway`
- `GatewayClasses`

#### 存储管理

- `PersistentVolumeClaims`
- `PersistentVolumes`
- `StorageClasses`

#### 安全管理

- `ServiceAccounts`
- `Roles`
- `ClusterRoles`
- `RoleBindings`
- `ClusterRoleBindings`

#### 配置管理

- `ConfigMaps`
- `Secrets`

#### 资源管理

- `HPAs`
- `VPAs`
- `ResourceQuotas`
- `LimitRanges`

### 4.3 一期明确不做

- 多集群
- Helm 管理
- 批量操作
- 在线 YAML 编辑器
- 自建第二套权限体系
- 数据库驱动的审计检索平台

## 5. 信息架构

### 5.1 左侧导航

| 一级导航 | 二级菜单 | 说明 |
| --- | --- | --- |
| 集群 | `Overview` `Namespaces` `Nodes` | 集群状态、命名空间、节点 |
| 资源全景图 | `Topology` | 资源关系画布 |
| 工作负载 | `Pods` `Deployments` `StatefulSets` `DaemonSets` `ReplicaSets` `Jobs` `CronJobs` | 工作负载资源列表与操作 |
| 网络管理 | `Services` `Endpoints` `Ingresses` `IngressClasses` `NetworkPolicies` `Gateway` `GatewayClasses` | 服务发现、入口与网络策略 |
| 存储管理 | `PersistentVolumeClaims` `PersistentVolumes` `StorageClasses` | 存储资源 |
| 安全管理 | `ServiceAccounts` `Roles` `RoleBindings` | RBAC 与身份 |
| 配置管理 | `ConfigMaps` `Secrets` | 配置与密钥 |
| 资源管理 | `HPAs` `VPAs` `ResourceQuotas` `LimitRanges` | 弹性与资源治理 |

### 5.2 导航约束

- 二级菜单统一保持英文资源名
- `工作负载` 不再单独设置 `Workloads` 二级菜单
- `Topology` 作为独立资源域页面，不嵌入 `Overview`
- 一级导航负责资源域分组，二级菜单负责具体资源类型

### 5.3 全局布局

- 左侧为资源域导航
- 顶部保留全局状态区
- 顶部状态区固定展示：
  - 当前命名空间
  - 当前登录身份
  - 当前接入模式
- 内容区按页面类型承载：
  - 总览页
  - 列表页
  - 详情页
  - 关系画布
  - 排障页

## 6. 页面设计基线

### 6.1 总体原则

- 不使用默认 `Ant Design` 后台模板式大卡片布局
- 页面结构应在数据增减时保持稳定，不因数据多少造成大面积跳动
- 信息密度要偏运维控制台，而不是营销看板
- 主次关系要明显：导航、顶部上下文、主体内容、局部操作分层清晰

### 6.2 登录页

- 提供 `ServiceAccount Token` 输入框
- 登录成功后进入 `/cluster/overview`
- 保留演示模式，方便仅查看前端骨架效果
- 真实联调以 `Bearer Token` 为主，演示模式不作为正式接入路径

### 6.3 Overview 页面

- 采用一页合成式总览
- 第一屏展示：
  - `Kubernetes Version`
  - `Nodes Ready / Total`
  - `Namespaces`
  - `Pods Running / Total`
  - `CPU Usage`
  - `Memory Usage`
- 同页展示：
  - 最近 `10` 条 `Warning` 事件
  - 节点状态概览
  - `Pod` 数量 Top 5 命名空间
- 避免重复展示同一批信息
- 结构优先稳定，不因局部数据为空而改变整体骨架

### 6.4 资源列表页

- 统一使用 `ProTable`
- 默认提供搜索、筛选、刷新、详情跳转
- 常见列基线：
  - 状态
  - 名称
  - 命名空间
  - 创建时间
  - 关键资源属性
- 危险操作统一进入操作列
- 对支持创建和编辑的资源，提供统一入口

### 6.5 Topology 页面

- 页面主角必须是拓扑画布，不做重卡片堆砌
- 右侧详情不固定常驻，按需显示
- 底部状态卡片和额外工作台不作为主结构
- 关系语义以真实 Kubernetes 对象关系为准
- 当前首批关系重点覆盖：
  - `Service -> Pod`
  - `Pod -> ReplicaSet`
  - `ReplicaSet -> Deployment`
- 切换命名空间后必须稳定刷新，不允许白屏
- 空命名空间、低数据量、无关系图谱都必须有稳定空态

## 7. 前端技术方案

### 7.1 技术栈

- `React`
- `TypeScript`
- `Vite`
- `React Router`
- `Ant Design`
- `Ant Design ProTable`
- `TanStack Query`
- `Zustand`
- `Axios`
- `Tailwind CSS`
- `@xyflow/react` 用于拓扑画布

### 7.2 前端设计原则

- 能直接使用 `Ant Design` 的场景优先使用 `Ant Design`
- 不把 `Ant Design` 默认后台风格当成最终 UI
- `TanStack Query` 负责服务端状态、缓存、刷新和错误重试
- `Zustand` 负责登录态、命名空间、UI 偏好和本地会话信息
- `Tailwind CSS` 只做补充，不替代组件体系
- 页面级渲染异常必须被错误边界隔离，不能导致整站白屏

## 8. 后端技术方案

### 8.1 技术栈

- `Go`
- `Gin`
- 单体服务
- `REST API`
- `WebSocket`
- `client-go`
- `informer/cache`

### 8.2 后端职责

- Bearer Token 登录校验
- 基于 Token 创建用户态 Kubernetes Client
- 聚合集群总览数据
- 提供资源列表、详情与关系图接口
- 代理日志流与终端会话
- 输出轻量审计日志

### 8.3 后端设计原则

- 读操作优先走缓存和聚合
- 写操作直接调用 `API Server`
- 第一阶段不引入数据库
- 接口按资源域组织，而不是按页面临时拼装
- 保持单体边界清晰，为后续拆分预留接口层和服务层

## 9. 认证、真实集群接入与审计

### 9.1 真实集群接入方式

- 后端运行在本机开发机
- 后端通过本地 `kubeconfig` 连接真实实验集群
- 前端通过登录页提交 `ServiceAccount Token`
- 后端使用该 Token 透传访问 Kubernetes API

### 9.2 kubeconfig 读取顺序

- `K8S_ADMIN_KUBECONFIG`
- `KUBECONFIG`
- `~/.kube/config`

### 9.3 Token 登录约束

- 登录接口仅负责校验 Token 可用性并返回基础上下文
- 后续请求统一通过 `Authorization: Bearer <token>` 访问
- 权限不足时直接回显 Kubernetes `RBAC` 结果，不做本地越权兜底

### 9.4 审计策略

- 第一阶段保留轻量审计
- 关键变更动作输出结构化日志到 `stdout`
- 后续是否落库、是否支持检索与回放，放到后续阶段再评估

## 10. 接口与数据契约

### 10.1 接口组织

- `/api/v1/auth/...`
- `/api/v1/overview/...`
- `/api/v1/nodes`
- `/api/v1/topology/...`
- 后续按资源域扩展：
  - `/api/v1/workloads/...`
  - `/api/v1/network/...`
  - `/api/v1/storage/...`
  - `/api/v1/security/...`
  - `/api/v1/config/...`
  - `/api/v1/resources/...`

### 10.2 响应信封

所有接口统一返回：

```json
{
  "code": "OK",
  "message": "success",
  "data": {}
}
```

### 10.3 数据契约约束

- 所有数组字段必须返回 `[]`，不能返回 `null`
- 命名空间范围的数据接口必须显式接收并处理 `namespace`
- 前端切换命名空间后必须触发重新拉取，不能继续复用旧数据
- 页面渲染不得直接假设数组非空，空态必须可渲染
