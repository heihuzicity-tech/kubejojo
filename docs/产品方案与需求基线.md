# kubejojo 产品方案与需求基线

## 1. 文档说明

- 本文档用于沉淀 `kubejojo` 当前已经确认的产品边界、信息架构、技术路线和已落地能力。
- 本版本以 `main` 分支在 `2026-04-24` 的实际代码状态为准，不再引用旧工作树或未合并分支作为“当前状态”。
- 文档会明确区分“已确认长期方向”和“当前代码已实现状态”，避免把规划目标和现状混写。
- 未单独展开的小问题，默认参考 `Kubernetes Dashboard` 与 `Headlamp` 的成熟实践：
  - 功能分组与资源域导航优先参考 `Headlamp`
  - 总览页、运维导向和信息密度优先参考 `Kubernetes Dashboard`
- 视觉实现不直接照抄官方产品，但交互优先服从清晰层级、稳定骨架、真实数据承载能力和运维效率。

## 2. 产品定位

- 项目名称：`kubejojo`
- 产品定位：面向企业内部运维场景的 `Kubernetes` 单集群管理系统
- 当前定位：`单集群增强版`
- 当前接入方式：
  - 开发态：本机启动前后端，直连真实实验集群
  - 发布态：前端静态资源内嵌到后端二进制，按单服务交付
- 当前认证方式：`ServiceAccount Bearer Token`
- 当前默认语言：中文
- 第一阶段不做 `i18n`
- 当前产品目标不只是“资源查看”，而是同时覆盖：
  - 统一资源导航
  - 真实集群状态总览
  - 资源列表与详情
  - YAML 读写与基础运维动作
  - 发布版本检查、安装、回滚、重启

## 3. 已确认的核心决策

- 权限模型：直接复用 `Kubernetes RBAC`
- 前端：`React + TypeScript + Vite + Ant Design`
- 状态管理：`TanStack Query + Zustand`
- 列表组件：优先使用 `Ant Design ProTable`
- 请求库：`Axios`
- 样式补充：`Tailwind CSS`
- 后端：`Go` 单体服务
- Web 框架：`Gin`
- Kubernetes SDK：`client-go`
- 接口风格：以 `REST` 为主
- 实时能力：`WebSocket` 用于 `Pod Exec Terminal`
- 数据存储：当前阶段不引入数据库
- 审计：保留轻量审计边界，现阶段以进程日志和动作返回为主
- 交付方式：
  - `source` 模式下前后端分离运行
  - `release` 模式下前端资源内嵌到后端二进制
- 更新方式：发布态支持基于 `GitHub Releases` 的版本检测、安装、回滚和重启
- 设计取向：保留 `Ant Design` 组件能力，但 `Layout / Navigation / Overview / List / Detail` 采用自定义骨架，不使用默认后台模板风格

## 6. 页面设计基线

### 6.1 总体原则

- 不使用默认 `Ant Design` 后台模板式大卡片布局
- 页面结构应在数据增减时保持稳定，不因数据多少造成大面积跳动
- 信息密度要偏运维控制台，而不是营销看板
- 主次关系要明显：导航、顶部上下文、主体内容、局部操作分层清晰
- 所有页面都必须兼容：
  - 真实数据
  - 空态
  - 权限不足
  - 接口失败时的回退展示

### 6.2 登录页

- 提供 `ServiceAccount Token` 输入框
- 登录成功后进入 `/cluster/overview`
- 保留演示模式，方便只查看前端骨架和交互
- 真实联调以 `Bearer Token` 为主，演示模式不作为正式接入路径

### 6.3 Overview 页面

- 采用一页合成式总览
- 第一屏固定展示：
  - `Kubernetes Version`
  - `Nodes Ready / Total`
  - `Namespaces`
  - `Pods Running / Total`
  - `CPU Usage`
  - `Memory Usage`
- 同页展示：
  - 最近 `10` 条 `Warning` 事件
  - 节点状态概览
  - `Pod` 数量 Top 5 命名空间
- 结构优先稳定，不因局部数据为空而改变整体骨架

### 6.4 资源列表页

- 统一使用 `ProTable`
- 默认提供搜索、刷新、详情跳转
- 常见列基线：
  - 状态
  - 名称
  - 命名空间
  - 创建时间
  - 关键资源属性
- 危险操作统一进入操作列
- 当前多数资源列表页已经接入：
  - YAML 查看
  - YAML 编辑
  - 删除确认
  - 常用创建入口

### 6.5 资源详情页

- 详情页按资源类型组织 `overview / yaml / related` 等标签页
- `Pod` 详情页额外保留：
  - `containers`
  - `events`
  - `logs`
  - `terminal`
- 详情页默认承担“查看一屏关键信息 + 进入 YAML/关联关系”的职责，不做重型工作台堆叠

### 6.6 Topology 页面

- 页面主角必须是拓扑画布，不做重卡片堆砌
- 右侧详情采用抽屉式或按需显示，不固定占位
- 关系语义以真实 Kubernetes 对象关系为准
- 当前重点覆盖：
  - `Service -> Pod`
  - `Pod -> ReplicaSet`
  - `ReplicaSet -> Deployment`
  - 以及工作负载、网络、存储相关关系
- 切换命名空间后必须稳定刷新，不允许白屏
- 空命名空间、低数据量、无关系图谱都必须有稳定空态

### 6.7 更新管理页

- 页面承担系统版本和更新动作的统一入口
- 需要清晰展示：
  - 当前版本状态
  - 发布信息
  - 权限限制
  - 安装、回滚、重启动作结果
- 长动作必须给出明确的状态反馈，不能让用户靠刷新猜测结果

## 7. 前端技术方案

### 7.1 技术栈

- `React 19`
- `TypeScript`
- `Vite`
- `React Router`
- `Ant Design`
- `Ant Design Pro Components`
- `TanStack Query`
- `Zustand`
- `Axios`
- `Tailwind CSS`
- `@xyflow/react` 用于拓扑画布
- `xterm` 用于 `Pod Exec Terminal`

### 7.2 前端设计原则

- 能直接使用 `Ant Design` 的场景优先使用 `Ant Design`
- 不把 `Ant Design` 默认后台风格当成最终 UI
- `TanStack Query` 负责服务端状态、缓存、刷新和错误重试
- `Zustand` 负责登录态、命名空间、会话模式和本地上下文
- `Tailwind CSS` 只做补充，不替代组件体系
- 页面级异常必须被错误边界隔离，不能导致整站白屏
- 在 `demo` 模式下允许安全回退展示，不把接口失败直接暴露成整页不可用

## 8. 后端技术方案

### 8.1 技术栈

- `Go`
- `Gin`
- 单体服务
- `REST API`
- `WebSocket`
- `client-go`
- 嵌入式静态资源交付

### 8.2 后端职责

- Bearer Token 登录校验
- 基于 Token 创建用户态 Kubernetes Client
- 聚合集群总览数据
- 提供资源列表、详情、YAML、关系图和运维动作接口
- 代理 `Pod Exec Terminal`
- 提供版本检查、安装、回滚、重启能力
- 校验更新相关主体权限
- 为 release 模式提供内嵌前端静态资源

### 8.3 后端设计原则

- 读取类接口尽量返回前端可直接消费的聚合结构
- 写操作直接调用 `API Server`，不引入本地状态真相源
- 第一阶段不引入数据库
- 接口按稳定资源能力组织，而不是按单个页面临时拼装
- 单进程下通过进程内系统操作锁避免更新、回滚、重启动作并发执行
- release 构建必须保证前端资源与后端版本信息一起交付

## 9. 认证、真实集群接入与审计

### 9.1 真实集群接入方式

- 后端运行在本机开发机
- 后端通过本地 `kubeconfig` 连接真实实验集群
- 前端通过登录页提交 `ServiceAccount Token`
- 后续请求通过 `Authorization: Bearer <token>` 访问后端接口

### 9.2 kubeconfig 读取顺序

- `KUBEJOJO_KUBECONFIG`
- `KUBECONFIG`
- `~/.kube/config`

### 9.3 Token 登录约束

- 登录接口负责校验 Token 可用性并返回基础上下文
- 权限不足时直接回显 Kubernetes `RBAC` 结果，不做本地越权兜底
- `demo` 模式仅用于前端安全演示，不代表真实认证成功

### 9.4 审计策略

- 当前阶段仍保持轻量实现
- 关键变更动作需要保留清晰的接口返回和服务日志
- 是否落库、是否支持检索与回放，仍放在后续阶段评估

- 当前主要接口族包括：
  - `/api/v1/auth/...`
  - `/api/v1/system/...`
  - `/api/v1/overview/...`

  - `/api/v1/ingresses`
  - `/api/v1/ingressclasses`
  - `/api/v1/networkpolicies`
  - `/api/v1/serviceaccounts`
  - `/api/v1/roles`

  - `/api/v1/rolebindings`
  - `/api/v1/clusterrolebindings`

  - `/api/v1/persistentvolumes`
  - `/api/v1/storageclasses`
  - `/api/v1/manifests`

### 10.2 响应信封

所有接口统一返回：

```json
{
  "code": "OK",
  "message": "success",
  "data": {}
}
```

### 10.3 数据契约约束

- 所有数组字段必须返回 `[]`，不能返回 `null`
- 命名空间范围的数据接口必须显式接收并处理 `namespace`
- 前端切换命名空间后必须重新拉取数据，不能继续复用旧数据
- 页面渲染不得直接假设数组非空，空态必须可渲染
- 长时动作接口必须给出明确错误信息，不能只返回泛化失败

- 资源创建能力已覆盖常用对象，但还没有做到“所有资源都可创建”
- 审计仍是轻量实现，没有独立审计存储和查询界面
- 批量操作、Helm、Gateway 相关能力仍未纳入当前产品范围
